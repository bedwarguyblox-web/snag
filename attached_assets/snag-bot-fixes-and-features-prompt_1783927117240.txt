================================================================================
PRIORITY 1 — DEALS HAVE NO ACTUAL CONVERSATION (the core reported bug)
================================================================================

THE BUG: When a deal starts, `_create_deal()` in cogs/deals.py DMs the buyer
and seller SEPARATELY. These are two independent, disconnected DM channels
(bot↔buyer and bot↔seller) — the two Discord users are total strangers who
likely share no mutual server and cannot message each other directly. Right
now the deal panel only has action buttons (Mark Complete / Cancel / Report).
There is no way for the two parties to actually talk to coordinate the trade
("meet at spawn", "sent it, check your chest", "what's your IGN again"). This
is the single most important fix in this pass.

THE FIX — build a message relay bridge between the two DMs for the duration
of an active deal:

1. Add an `on_message` listener (in cogs/deals.py, registered via
   `@commands.Cog.listener()` on the Deals cog) that:
   - Ignores messages from bots (`message.author.bot`) and anything that
     isn't a DM (`isinstance(message.channel, discord.DMChannel)`).
   - Looks up whether `message.author.id` is the initiator OR seller on any
     Deal with `status == "active"` (there can only ever be one — see the
     existing "one active deal per user" gate in `_create_deal`, condition
     (c) — so this lookup is always unambiguous).
   - If no active deal is found for that user, return immediately and do
     NOT reply (no nagging pointer message — keep it silent so we don't spam
     people who just DM the bot randomly; the How-To-Use button is the
     right place for guidance, not on_message).
   - If an active deal IS found, resolve the *other* party's user ID, fetch/
     get their DM channel (`bot.get_user(other_id) or await bot.fetch_user(other_id)`
     then `.send(...)`), and relay the message with a clear author prefix,
     e.g.:
       "💬 **{message.author.display_name}:** {message.content}"
     If the original message has attachments, relay their `.url`s too (a DM
     the bot receives is exempt from needing the Message Content privileged
     intent — confirmed current Discord policy: this exemption applies to
     "DMs that it receives", so plain `Intents.default()` — which the bot
     already uses — is sufficient; you do NOT need to enable the
     Message Content privileged intent for this feature).
   - Wrap the relay `.send()` in try/except for `discord.Forbidden` — if it
     fails, DM the ORIGINAL sender back: "⚠️ Couldn't deliver that — the
     other party's DMs may be closed. Try Report Issue if this keeps
     happening."
   - On every successful relay, update `Deal.last_activity_at =
     datetime.now(timezone.utc)` for that deal (open your own session; do
     NOT reuse a session across the on_message handler and unrelated code).
     This is important: right now `last_activity_at` is ONLY bumped by
     button clicks (mark_complete_btn), so two people actively chatting
     but never clicking a button would incorrectly auto-expire at the 48h
     mark even though they're clearly active. Relayed messages must count
     as activity.
   - Skip relaying anything that looks like it might be a command accident
     (e.g. starts with "!" or "/") is NOT necessary — this bot is slash-only,
     there's no prefix-command collision risk. Relay everything.

2. Update `build_deal_panel_embed()` in utils/embeds.py to make the very
   first line of the description explicitly tell people they can just type:
       "💬 Just type in this DM to talk to the other trader — I'll pass
       your messages along."
   This is the single biggest UX fix here: right now nothing in the panel
   even hints that typing does anything.

3. This relay only runs while `Deal.status == "active"`. Once a deal is
   completed/cancelled/expired, further DMs from either party should NOT be
   relayed (the lookup already naturally excludes non-active deals — just
   make sure the DB filter is `Deal.status == "active"`, not "any deal").

PRIORITY 2 — DEAL LIFECYCLE EVENTS DON'T NOTIFY THE OTHER PARTY
================================================================================

THE BUG (confirmed by reading cogs/deals.py): `cancel_btn()` only ever calls
`interaction.followup.send(...)`, which is ephemeral and visible ONLY to
whoever clicked Cancel. The other party is never told the deal was cancelled
— their copy of the deal panel just sits there looking active forever, and
if they click a button on it later they'll get a confusing "deal no longer
active" error out of nowhere. Same root problem, smaller version, on
`mark_complete_btn()`: when only ONE party confirms, the other party is never
nudged that their counterpart already confirmed and it's their turn.

Also confirmed: NONE of the three deal-ending code paths (`cancel_btn`,
`mark_complete_btn`'s both-confirmed branch, and `_expire_deal()` in
background_tasks.py) ever edit the original persistent deal-panel DM messages
(`deal.dm_message_id_initiator` / `deal.dm_message_id_seller`, both already
stored on the Deal row). They just leave the old panel sitting there with
live-looking buttons that will error out if clicked. Close it out properly
every time.

THE FIX:

1. Add one shared helper in cogs/deals.py:

     async def _close_deal_panels(bot, deal, closing_embed: discord.Embed) -> None:
         """Edit both parties' stored DM panel messages to a closed state,
         removing the view. Called from every deal-termination path. Never
         raises — wrap each edit in its own try/except for discord.NotFound
         (message deleted by the user) and discord.Forbidden."""

   For each of (deal.initiator_id, deal.dm_message_id_initiator) and
   (deal.seller_id, deal.dm_message_id_seller): fetch the user, fetch/open
   their DM channel, `.fetch_message(id)` then `.edit(embed=closing_embed,
   view=None)`. If dm_message_id is None (DM failed originally) or fetch
   raises NotFound/Forbidden, just skip that side — log at debug level,
   don't raise.

2. In `cancel_btn()`:
   - After committing `status = "cancelled"`, DM the OTHER party (not the
     clicker — they already got their ephemeral confirmation) a clear
     message: "❌ **{clicker display name}** cancelled Deal #{id} for
     **{listing.title}**. The listing has been reopened — no action needed
     from you." Wrap in try/except discord.Forbidden.
   - Call `_close_deal_panels(bot, deal, <a "🔒 Cancelled" embed>)`.

3. In `mark_complete_btn()`:
   - Partial-confirm branch (only one side confirmed so far): DM the OTHER
     party: "✅ **{name}** marked Deal #{id} as complete on their end!
     Click **Mark Complete** on your side too to finish the trade." Wrap in
     try/except discord.Forbidden. Do this in ADDITION to the existing
     ephemeral "waiting for the other party" message to the clicker — both
     should happen.
   - Both-confirmed branch (deal completes): after `_send_review_prompts`,
     also call `_close_deal_panels(bot, deal, <a "✅ Completed" embed>)` so
     the original panel doesn't linger with working-looking buttons next to
     the separate review-prompt DM.

4. In `_expire_deal()` (background_tasks.py): the existing "Deal Auto-Closed"
   DM to both parties already exists and is fine — just ALSO call
   `_close_deal_panels(bot, deal, <a "⏱️ Expired" embed>)` right after (import
   it locally inside the function to avoid a circular import, same pattern
   already used elsewhere in this file for `DealPanelView`).

5. Personalize `build_deal_panel_embed()` instead of building one generic
   embed and sending it to both parties (current behavior in `_create_deal`
   and `_resolve_auction` — one `deal_embed` object built once, sent twice).
   Change its signature to something like:

     build_deal_panel_embed(deal, listing, *, viewer_role: str,
                             counterpart_user, counterpart_profile,
                             color=None)

   and build it TWICE per deal-start (once per recipient) so each side's
   panel clearly shows:
     - "Your role: 🛒 Buyer" or "💰 Seller"
     - Who they're trading with (mention + IGN + rating), reusing the same
       IGN/rating formatting already in `build_listing_embed`
     - The agreed price (`listing.price`, or "current bid" for auctions —
       reuse the same price-string logic already in `build_listing_embed`
       so the two don't drift out of sync)
   Update the three call sites (`_create_deal`, `_resolve_auction`, and
   anywhere else a deal panel is sent) to fetch both users'
   UserProfile rows (`get_or_create_profile`) before building either embed —
   `_create_deal` already fetches both Discord user objects, just extend it
   to also fetch both profiles.

PRIORITY 5 — BUGS FOUND ON INSPECTION (fix all of these — you were asked to
be the inspector, these are real, confirmed issues, not speculative ones)
================================================================================

1. SILENT COMPONENT-ERROR SWALLOWING — none of the View subclasses in this
   codebase (ListingActionView, DealPanelView, FilterView, every wizard step
   view, PaginatorView, SetupView, etc.) override `on_error`. discord.py's
   default behavior on an uncaught exception inside a button/select callback
   is to print a traceback to stderr and tell the user NOTHING — from their
   side the interaction just silently fails ("This interaction failed").
   Fix: add one small base class, e.g. in a new utils/base_view.py:

     class SnagView(discord.ui.View):
         async def on_error(self, interaction, error, item):
             logger.exception("Unhandled error in %s (item=%s)",
                               type(self).__name__, item, exc_info=error)
             msg = ("⚠️ Something went wrong on my end. Please try again, "
                    "and let a server admin know if it keeps happening.")
             try:
                 if interaction.response.is_done():
                     await interaction.followup.send(msg, ephemeral=True)
                 else:
                     await interaction.response.send_message(msg, ephemeral=True)
             except Exception:
                 pass

   Then change every `class X(discord.ui.View)` in the codebase to
   `class X(SnagView)`. This is a mechanical, low-risk change — do it across
   every file, it's the single highest-value reliability fix in this pass.

2. UNNECESSARY PRIVILEGED INTENT — main.py sets `intents.members = True`
   (the privileged GUILD_MEMBERS intent). The only place that appears to
   need it is `is_admin()` in utils/checks.py, which does
   `interaction.guild.get_member(interaction.user.id)` to check
   `.guild_permissions` and `.roles`. That's unnecessary: in a guild
   interaction, `interaction.user` is ALREADY a full `discord.Member` object
   (Discord includes complete member data — permissions and roles included —
   directly in every interaction payload), so `interaction.user
   .guild_permissions.manage_guild` and `interaction.user.roles` work
   without any member cache or intent at all. Fix `is_admin()` to use
   `interaction.user` directly instead of calling `guild.get_member(...)`,
   then remove `intents.members = True` from main.py entirely. This matters
   because Discord's Privileged Intent review is tied to your app's total
   user count (10,000+ users triggers a review requirement, per Discord's
   current developer policy as of mid-2026) — the fewer privileged intents a
   "global, meant to scale to many servers" bot declares, the smoother that
   review stays as Snag grows. Also remove `intents.guild_messages = True` —
   grep the codebase; nothing currently listens for guild-channel
   `on_message` events (the new Priority-1 relay only touches DMs, which are
   already covered by `Intents.default()`), so that line is dead weight too.

3. PER-GUILD COMMAND SYNC ON EVERY SINGLE BOOT — `setup_hook()` in main.py
   loops over EVERY guild the bot is in and calls `bot.tree.copy_global_to`
   + `bot.tree.sync(guild=guild)` on every single restart. Fine at a handful
   of servers; becomes a slow, rate-limit-risking startup routine once Snag
   is actually installed on many servers (the explicit "global, cross-server"
   goal of this bot). Fix: keep the ONE global `bot.tree.sync()` call on
   every boot (that's correct and necessary), but move the per-guild instant
   sync into `on_guild_join()` ONLY — so a newly-joined server gets its
   commands instantly, without re-syncing every server Snag has ever joined
   on every restart.

4. SQLITE DATA-LOSS RISK — CRITICAL, VERIFY BEFORE ANYTHING ELSE IN THIS
   SECTION. `snag-bot/database/snag.db` is listed in .gitignore. If this
   Repl's deployment does a fresh checkout/redeploy from git (rather than
   reusing the same persistent filesystem), every listing, deal, review,
   ban, and profile gets silently wiped. This is not hypothetical — this
   exact pattern (gitignored SQLite file wiped on redeploy) was already hit
   and documented as a gotcha in this same builder's PREVIOUS Discord bot
   project. Before doing anything else:
     a. Confirm with the current Replit deployment type (Reserved VM /
        Autoscale / whichever is configured) whether the working directory
        persists across redeploys, or if it does a fresh git checkout each
        time.
     b. If it does NOT reliably persist: either (i) move the SQLite file to
        a location Replit guarantees is persistent and document that path
        clearly in replit.md the same way the previous project did with
        DATABASE_PATH, or (ii) migrate to Replit's built-in PostgreSQL
        database integration (ask the Agent to add one — it's a first-class
        Replit feature: persistent, 20GB free, provisions a DATABASE_URL
        automatically) so this entire class of bug becomes structurally
        impossible instead of something to remember to avoid. Given this bot
        is explicitly meant to hold irreplaceable cross-server trading
        history and reputation data, option (ii) is the safer long-term
        call, but either is acceptable — the one unacceptable outcome is
        leaving this unverified.
     c. Fix the stale comment in snag-bot/.env.example — it currently says
        "DATABASE_URL is auto-provided by Replit — do not set it manually"
        but engine.py never reads DATABASE_URL at all (it hardcodes a local
        SQLite path). That comment is either leftover from an earlier plan
        or flat-out wrong right now — make the file match reality, whichever
        way (a)/(b) above land.

5. `/moderation stats` (cogs/moderation.py) reports GLOBAL counts (every
   listing/deal across every server) under a command that's gated as
   guild-admin-only and framed as "for this server." Either make the counts
   guild-scoped (filter by `Listing.origin_guild_id == interaction.guild_id`
   for the listings count; deals are trickier to scope by guild since a deal
   spans two possibly-different origin servers — at minimum scope the
   listings number, and either drop or clearly re-label the deal counts as
   "network-wide") or rename the command/description to make clear it's
   showing bot-wide numbers on purpose. Low priority — fix if time allows.

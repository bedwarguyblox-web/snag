# Snag Bot — Fix Prompt for Replit + Prevention Playbook

Two parts here:

1. **A ready-to-paste prompt** for Replit's agent that fixes all 7 confirmed bugs, in the right order, with the destructive step guarded.
2. **A prevention playbook** — why these specific bugs happened, and concrete habits/techniques (agent-specific and general) to keep them from recurring.

I cross-checked every bug against your actual code (not just the report) before writing this, and found one thing worth flagging up front: **the "Trusted Seller" bug is worse than it looks.** It's not just that nothing sets `_completed_deals` — there is no `completed_deals` column on `UserProfile` at all. It's a schema gap, not a logic gap, which means it has to be fixed alongside the migration in bug #1, not as a standalone code patch.

## Part 2 — Why these bugs happened, and how to stop the pattern

Looking across all 7, they split into three repeating failure modes — not seven unrelated mistakes:

- **Schema drift**: the model file says one thing, the live database says another, and nothing checks that they agree. (#1, and secretly #6's `completed_deals` gap)
- **Write path never built, read path assumes it exists**: something reads a flag/column/counter that nothing ever sets. (#1's badge, #3)
- **A safety pattern exists in one file but wasn't copied to its siblings**: cooldowns, try/except guards. (#2, #4)

Fixing the seven bugs doesn't fix the pattern. Here's what does, roughly in order of leverage.

### 1. Give schema changes a real migration path
`create_all()` only creates *missing* tables — it will never fix a column type, add a column to an existing table, or backfill data. That "safe on every startup" comment in `engine.py` is true and misleading at the same time: safe, but silently does nothing when you actually needed a change to take effect. This is the single highest-leverage fix here, because it caused bug #1 and would've silently swallowed the `completed_deals` column too if it had been added without a real migration.

- Adopt **Alembic** even for a small SQLite project — it turns "did the fix actually apply" from a question you find out about via a bug report into something `alembic upgrade head` either does or fails loudly on.
- If Alembic is overkill for now, at minimum add a tiny **schema version check at startup**: compare `PRAGMA table_info(table)` against `Base.metadata` for each table and log a hard `ERROR` (not silence) if they've diverged. That alone would have turned this bug into a startup log message instead of a runtime crash discovered by a user.
- Treat "delete the db and let it regenerate" as a **dev-only escape hatch**, never a fix you ship instructions for without first asking whether the target database has real data. (I built that check into Step 0 of the prompt above for exactly this reason.)

### 2. Every "flag" needs a traced write path before you trust it
`_completed_deals` and `global_feed_channel_id` are the same bug shape: a value is read somewhere, formatted nicely, gated behind a threshold or an `or` fallback — and looks completely implemented. Nothing about reading the code at the call site tells you it's dead; you only find out by tracing backward to see if anything ever *writes* it.

A cheap habit that catches this class of bug specifically:
- When you (or an agent) finish a feature that reads a model field or config value, explicitly grep for every place that *sets* it. If the answer is "nowhere else," that's your signal — either wire it up now or leave a `# TODO: no write path yet` comment so it doesn't quietly ship as "done."
- Periodically ask an agent to do exactly what generated your bug report: **"trace every flag and config field end-to-end from write path to read path and tell me which ones are read but never written."** That's a genuinely good recurring audit, not a one-time cleanup — worth doing again after a few weeks of new features, not just once.

### 3. When you add a guard in one place, grep for its siblings
The hex-color bug happened because `listings.py` and `deals.py` both correctly wrap `int(color, 16)` in try/except, but `admin_setup.py` — which does the exact same parse, twice — doesn't. The safe pattern already existed in the codebase; it just wasn't propagated everywhere the same operation happens.

- Whenever you (or an agent) add a `try/except`, a cooldown, a validation check, or any other guard, grep the repo for other call sites doing the *same kind of operation* (not just the same function name) and confirm they're consistent. `int(*.lstrip("#"), 16)` is a good example — a plain function-name search wouldn't have caught the inconsistency, but a search for that literal pattern would.
- **Validate at the boundary, defend at every consumption site.** Fixing the modal stops *new* bad values, but guilds that already saved `#GGGGGG` before the fix would still crash `preferences()` and panel sending. Defense in depth — validate on input, and never let a raw unchecked string reach `int(..., 16)` — is why Step 5 above fixes both.

### 4. Test the failure mode, not just the happy path
None of these bugs would've survived a couple of targeted tests:
- An insert-and-fetch test on `Listing` would have caught the PK bug immediately, in CI, instead of in production.
- A fuzz test on the hex modal with `"#GGGGGG"`, `"####"`, `""`, `"red"` would have caught the validation gap in seconds.
- A "create two listings within 5 minutes" test would have caught the missing cooldown.

You don't need a huge suite — a handful of tests aimed specifically at "what's the worst plausible input/timing here" catches most of what a broad manual read-through misses. If Replit supports a "Run tests" step, wire pytest into it so agent-written code gets checked mechanically before you're the one who finds the bug in production.

### 5. Use (and grow) the memory file this repo already has
You already have `.agents/memory/MEMORY.md` and a `sqlite-autoincrement-pk.md` entry documenting the exact PK-type lesson from earlier — that's a good pattern, and it's underused. The `create_all()`-doesn't-migrate lesson and the "config fields need a confirmed write path" lesson from this pass belong in there too (I added that as the last instruction in the Step-0 block of the prompt above). The value of this isn't just documentation — it's that a future agent session, or a future you, gets to start from "we already know not to do this" instead of rediscovering it by shipping the same bug shape again.

### 6. A short checklist for reviewing any agent-generated diff going forward
Whether it's Replit, Claude, or anyone else writing the code:

- **New model field or column?** → Does it ship with a migration, or does it silently rely on `create_all`?
- **New config value or flag?** → Is there a confirmed call site that writes it, not just one that reads it?
- **New guard/validation/cooldown?** → Did you grep for every sibling call site doing the same kind of operation?
- **New try/except added?** → Does the *matching* operation elsewhere in the codebase have the same guard?
- **Destructive fix suggested** (drop table, delete file, reset state)? → Has anyone confirmed there's no real data at stake first?
- **"It's implemented" claim** → Was it actually exercised end-to-end (real insert, real render, real failure case), or does it just read cleanly?

None of this is specific to Discord bots or SQLite — it's the same handful of questions worth asking of any AI-assisted change, since the bugs an agent introduces tend to look like exactly this: locally correct code that's globally inconsistent with a sibling file, or with the live state of a database it can't see.

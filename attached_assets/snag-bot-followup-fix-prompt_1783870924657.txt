BUG A — Pagination footer grows every time a user revisits a page
═══════════════════════════════════════════════════════════════════
File: snag-bot/utils/pagination.py, PaginatorView.current_page_embed()
(around lines 50-54)

What's there now:
    if isinstance(page_items[0], discord.Embed):
        embed = page_items[0]
        existing_footer = embed.footer.text or ""
        embed.set_footer(text=f"{existing_footer}  •  {self.page + 1}/{self.total_pages}")
        return embed

Why it's broken: `self.items` is built once in __init__ and reused for the
life of the view, so `page_items[0]` is the SAME Embed object every time a
given page is rendered — not a fresh copy. `existing_footer` reads whatever
is CURRENTLY in the footer, which on the second visit to a page is already
the mutated value from the first visit. Each revisit appends another
"  •  N/M" suffix instead of replacing it.

I confirmed this by simulating Prev/Next clicks:
  initial render (page 0): footer='Listing #1 • Active  •  1/3'
  click Next -> page 1:    footer='Listing #2 • Active  •  2/3'
  click Prev -> page 0:    footer='Listing #1 • Active  •  1/3  •  1/3'
  click Next -> page 1:    footer='Listing #2 • Active  •  2/3  •  2/3'
It keeps growing with every click on a previously-seen page for as long as
the view is alive (up to its 120s timeout).

Fix: don't derive the new footer from the embed's current (possibly already
mutated) footer text. Capture each embed's ORIGINAL footer once, before any
page-suffix is ever appended, and always build the displayed footer from
that saved original — e.g. store `original_footer` per item the first time
current_page_embed() touches it (or, cleaner, store the original footer
text alongside each embed when the list is built in listings.py, so
pagination.py never has to infer it from possibly-mutated state). The
displayed footer should always be exactly "<original> • N/M", never
"<original> • N/M • N/M • N/M...".

Verification: build a PaginatorView with 2+ listing embeds, click Next then
Prev then Next then Prev several times, and confirm the footer text is
identical every time you land on the same page — it should never grow.

BUG B — Review double-submit guard likely raises a different crash
         instead of the friendly error it's supposed to show
═══════════════════════════════════════════════════════════════════
File: snag-bot/cogs/reviews.py, around lines 92-125

What's there now:
    async with AsyncSessionLocal() as session:
        async with session.begin():
            ...
            session.add(review)
            try:
                await session.flush()
            except IntegrityError:
                await session.rollback()
                await interaction.followup.send(
                    embed=build_error_embed("You've already submitted a review for this deal."),
                    ephemeral=True,
                )
                return
            ...

Why this is likely still broken: `session.rollback()` is called manually
INSIDE an active `async with session.begin():` block. That block's own
__aexit__ still expects to own the commit/rollback of this transaction —
when the `return` above causes the `with` block to exit normally (no
exception propagating, since we already caught it), SQLAlchemy tries to
commit a transaction that was already manually rolled back out from under
it. Mixing manual session.rollback()/commit() with an active session.begin()
context manager is a documented SQLAlchemy async pitfall and typically
raises its own error at that point (transaction already closed / pending
rollback), which would replace the friendly "already reviewed" message with
an unhandled crash — in exactly the race-condition case this fix was
written to catch.

I was not able to install SQLAlchemy in my review sandbox to execute this
end-to-end, so I can't give you a 100% confirmed stack trace — but the
pattern itself is wrong regardless of the exact error text, and it should be
fixed rather than left to find out in production.

Fix: don't catch IntegrityError inside the session.begin() block at all.
Let it propagate out of the block — session.begin()'s own exception path
will correctly roll back — and catch it OUTSIDE, at the same level as the
`async with AsyncSessionLocal()`:

    try:
        async with AsyncSessionLocal() as session:
            async with session.begin():
                ...
                session.add(review)
                await session.flush()   # let IntegrityError propagate
                ...
    except IntegrityError:
        await interaction.followup.send(
            embed=build_error_embed("You've already submitted a review for this deal."),
            ephemeral=True,
        )
        return

Verification: this only triggers on the genuine race (two near-simultaneous
submissions that both pass the earlier "existing review" SELECT check before
either commits), so a normal manual double-click won't exercise it — the
existing SELECT check catches that case first. To actually test the fix,
simulate the race directly: open two sessions, insert two Review rows with
the same (deal_id, reviewer_id) back-to-back without the SELECT guard in
between (or add a short asyncio.sleep between the SELECT and the flush in a
test copy of the code to widen the race window), and confirm the SECOND
insert produces the friendly error message and NOT an unhandled exception in
the logs.

STEP 1 — Price sort filters (Low→High / High→Low)
═══════════════════════════════════════════════════════════════════
File: cogs/listings.py, class FilterView (around line 582) and
_run_listing_query (around line 658).

FilterView's row 4 currently has exactly 3 buttons (🔍 Search,
🏠 Server Listings, 🌐 Global Listings) — there's room for exactly 2 more
in that row before it's full. Add:

    @discord.ui.button(label="💰 Price: Low→High", style=discord.ButtonStyle.secondary, custom_id="filter:sort_low", row=4)
    async def sort_low(self, interaction: discord.Interaction, _):
        await interaction.response.defer(ephemeral=True)
        self._filters["sort"] = "price_asc"
        await _run_listing_query(interaction, self._filters, guild_id=self._filters.get("_last_guild_id"))

    @discord.ui.button(label="💰 Price: High→Low", style=discord.ButtonStyle.secondary, custom_id="filter:sort_high", row=4)
    async def sort_high(self, interaction: discord.Interaction, _):
        await interaction.response.defer(ephemeral=True)
        self._filters["sort"] = "price_desc"
        await _run_listing_query(interaction, self._filters, guild_id=self._filters.get("_last_guild_id"))

(The `_last_guild_id` bit only matters if you want sort to be re-clickable
after a Server Listings search without losing the guild scope — check how
guild_id currently flows into repeated filter clicks and match the existing
convention; don't invent a new state-tracking mechanism if one already
exists.)

In _run_listing_query, after the query is built and before `.limit(...)` is
applied, add ordering based on the sort filter:

    sort = filters.get("sort")
    if sort == "price_asc":
        q = q.order_by(Listing.price.asc())
    elif sort == "price_desc":
        q = q.order_by(Listing.price.desc())

Listings with a NULL price (if that's possible in this schema — check
Listing.price nullability) should sort last regardless of direction, not
first — confirm SQLite's default NULL ordering does what you want here, or
add `.order_by(Listing.price.is_(None), ...)` ahead of the asc/desc clause
if not.

STEP 2 — Give PaginatorView a per-page action button (needed by #3 and #4)

Right now PaginatorView only has Prev/Next. Both "Start Deal on the listing

of per-instance.

Add an optional `action_buttons` param to PaginatorView.__init__ — a list
(same length/order as `items`) where each entry is either None (no action
button on that page) or a dict describing 1-2 buttons to show for that item,

"callback": some_async_fn}`. Store it, and add a second row of up to 2
button slots that get relabeled and rebound to the current page's callback
every time _update_buttons() runs (same place Prev/Next disabled-state
already gets refreshed) — e.g.:

        ...
        self._action_buttons = action_buttons  # one entry per item, aligned by index
        self._update_buttons()

    def _update_buttons(self) -> None:
        self.prev_btn.disabled = self.page == 0
        self.next_btn.disabled = self.page >= self.total_pages - 1
        self._refresh_action_buttons()

        # to whatever the CURRENT page's item needs, mirroring how
        # ListingActionView swaps label/custom_id in __init__.
        start = self.page * self.per_page
        configs = self._action_buttons[start] if self._action_buttons else None
        ...

Don't put deals-specific logic (calling _create_deal, opening PlaceBidModal,
etc.) inside pagination.py — utils/ shouldn't import from cogs/. Instead,
have each button's actual click handler be a plain callback function that
gets passed in from wherever PaginatorView is constructed (listings.py),
using the same local/deferred-import pattern already used in this codebase
(see cogs/listings.py line 542, `from cogs.deals import ListingActionView`

should stay generic; listings.py supplies the deal-specific or edit-specific

Test this in isolation before moving on: build a PaginatorView with 3 fake

bug last time. Get this right here; #3 and #4 both build on it.

STEP 3 — Add Start Deal / Place Bid buttons inside Check Listings
═══════════════════════════════════════════════════════════════════
File: cogs/listings.py, _run_listing_query (around line 714)

Currently: `embeds = [build_listing_embed(listing, profile) for listing, profile in rows]`
— this only has the embed, not the listing_id/format needed to wire an
action button. Change this to build BOTH the embeds list and a parallel
action_buttons list, one entry per row, using the exact same logic
ListingActionView.__init__ already uses to decide button label/behavior
(cogs/deals.py lines 31-45):

    embeds = []
    action_buttons = []
    for listing, profile in rows:
        embeds.append(build_listing_embed(listing, profile))  # keep existing call
        if listing.format == "auction":
            action_buttons.append({
                "label": "🔨 Place Bid",
                "style": discord.ButtonStyle.success,

        else:
            action_buttons.append({

    paginator = PaginatorView(embeds, action_buttons=action_buttons)

Watch the classic Python closure-in-a-loop bug here — capture listing_id as
a default argument (`lid=listing.listing_id`) like the sketch above, not by
referencing the loop variable directly, or every button will fire for
whichever listing happened to be last in the loop.

For the direct-sale case, _create_deal (cogs/deals.py line 90) currently
assumes interaction.response was already deferred by the caller (see
ListingActionView.start_deal_btn, which calls
`interaction.response.defer(ephemeral=True)` before calling _create_deal).
Whatever calls _create_deal from the paginator button must do the same
defer first — write a small wrapper (`_start_deal_from_paginator` in the
sketch above) rather than calling _create_deal directly, so this doesn't
get missed.

Note: _create_deal already refuses a seller starting a deal on their own
listing (cogs/deals.py line ~157, "You can't start a deal with yourself.")
— that's an existing guard, don't duplicate it in the button rendering.

Add a 4th persistent button to MainPanelView (cogs/panel_views.py, after

        label="📋 My Listings",
        style=discord.ButtonStyle.secondary,

Add `start_my_listings` in cogs/listings.py, following the same shape as
start_check_listings — query the caller's own active listings:

    async def start_my_listings(interaction: discord.Interaction):
        async with AsyncSessionLocal() as session:
            result = await session.execute(
                select(Listing).where(
                    Listing.seller_id == interaction.user.id,
                    Listing.status == "active",
                )
            )
            rows = result.scalars().all()

        if not rows:
            await interaction.followup.send(
                embed=discord.Embed(title="No active listings", description="You don't have any active listings right now.", color=discord.Color.greyple()),
                ephemeral=True,
            )
            return

        embeds = [build_listing_embed(listing) for listing in rows]
        action_buttons = [
            [
                {"label": "✏️ Edit", "style": discord.ButtonStyle.primary,
                 "callback": lambda i, l=listing: i.response.send_modal(EditListingModal(l))},
                {"label": "🗑️ Cancel", "style": discord.ButtonStyle.danger,
                 "callback": lambda i, lid=listing.listing_id: _cancel_listing_from_paginator(i, lid)},
            ]
            for listing in rows
        ]
        paginator = PaginatorView(embeds, action_buttons=action_buttons)
        await interaction.followup.send(embed=paginator.current_page_embed(), view=paginator, ephemeral=True)

This is exactly why STEP 2's action_buttons needs to support up to 2 buttons
per page, not just 1 — My Listings needs Edit AND Cancel side by side,
Check Listings only needs one. Same closure-capture caution as STEP 3
applies to `l=listing` and `lid=listing.listing_id` here.

Reuse EditListingModal exactly as it exists today (cogs/listings.py line
866) — it already takes a full Listing object and already re-validates
ownership + active status inside on_submit, so no changes needed there.

For _cancel_listing_from_paginator, factor out the actual cancel logic
already written inline in the /listing cancel command (cogs/listings.py
lines 831-853) into a small reusable function both the slash command and
this new button call, instead of duplicating the ownership/status checks
in two places. Keep /listing cancel and /listing edit working exactly as
they do now — this step adds a button-driven alternative, it doesn't
replace the slash commands.

Since My Listings only shows status == "active" listings, both action
buttons only ever appear on listings that are actually editable/cancellable
under the existing rules (listing_edit already checks status == "active"),
so you shouldn't hit "can't edit a completed listing" in this flow at all
— but keep the modal's own re-check as a safety net regardless, in case a
listing's status changes between opening My Listings and clicking Edit.
